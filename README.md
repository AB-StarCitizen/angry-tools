# Star Citizen Game Data Parser & Database Sync (v4.8)

A comprehensive Python toolset designed to parse, normalize, and sync data-mined JSON records from *Star Citizen* game files (extracted from PKG files using [StarBreaker](https://github.com/diogotr7/StarBreaker)) into SQLite and PostgreSQL databases.

The system filters out non-pledge (AI, NPC, and test templates) ships to retain only player-ownable spacecraft and ground vehicles. It also maps nested slot hardpoint layouts, whitelists functional component types, normalizes grades/classifications, fetches friendly names from the Erkul.games API, and generates Markdown and JSON catalogs.

---

## 🚀 Features

* **Dual Database Support**: Fully synchronizes data to both local SQLite (`sc_database.db`) and remote PostgreSQL databases.
* **Pledge Ship Filtering**: Excludes NPC-only/AI templates and limits the inventory to player-pledgeable ships and vehicles (193 unique templates) by cross-referencing vehicle showroom display parameters.
* **Nested Loadout Parsing**: Recursively traverses ship port structures to map sub-attached components (e.g. weapons attached to gimbal mounts/turrets or missiles attached to racks).
* **Incremental Synchronization**: The PostgreSQL builder (`build_sc_db_pg.py`) uses a file sync metadata table to inspect file modification times (`mtime`) and sizes, skipping unchanged JSON files to maximize builder performance.
* **Erkul API Integration**: Translates developer class names (e.g., `GLSN_LaserRepeater_S3`) into player-facing friendly names (`Tormenter S3`), assigns item classifications, and maps numeric grades (1–4) to letter grades (A–D).
* **Automatic Orphan Cleanup**: Detects hardpoint layout modifications and clears outdated entries before inserting a ship's updated layout.
* **Catalog & Report Generation**: Automatically exports database query results into structured markdown documents (`pledge_ships_report.md`, `ship_default_loadouts.md`) and JSON configurations (`ship_default_loadouts.json`).

## 🗄️ Database Schema

The SQLite and PostgreSQL databases share a normalized, relational structure. The PostgreSQL tables utilize UUID primary keys generated by the `uuid-ossp` extension, whereas the SQLite schema uses autoincrement integer primary keys.

```mermaid
erDiagram
    SHIPS {
        UUID_or_INT id PK
        VARCHAR class_name UNIQUE
        VARCHAR display_name
        TEXT description
        VARCHAR career
        VARCHAR role
        VARCHAR manufacturer
    }

    COMPONENTS {
        UUID_or_INT id PK
        VARCHAR class_name UNIQUE
        VARCHAR name
        VARCHAR type
        VARCHAR sub_type
        INTEGER size
        VARCHAR grade
        VARCHAR manufacturer
        VARCHAR classification
    }

    SHIP_PORTS {
        UUID_or_INT id PK
        UUID_or_INT ship_id FK
        VARCHAR port_name
        VARCHAR parent_port
        INTEGER min_size
        INTEGER max_size
        VARCHAR accepted_types
        VARCHAR default_component_class
    }

    SHIP_DEFAULT_LOADOUTS {
        UUID_or_INT id PK
        UUID_or_INT ship_id FK
        VARCHAR port_name
        VARCHAR parent_port
        UUID_or_INT component_id FK
    }

    FILE_SYNC_METADATA {
        VARCHAR file_path PK
        DOUBLE_PRECISION last_modified
        BIGINT file_size
    }

    SHIPS ||--o{ SHIP_PORTS : "has"
    SHIPS ||--o{ SHIP_DEFAULT_LOADOUTS : "has"
    COMPONENTS ||--o{ SHIP_DEFAULT_LOADOUTS : "equipped_as"
```

### Table Definitions

1. **`ships`**: Stores core metadata for spaceships and ground vehicles.
2. **`components`**: Stores attaching parts, weapons, and systems. Filtering rules skip placeholder items (containing `"loc_placeholder"`).
3. **`ship_ports`**: Represents available item slots on a ship. Supports nested parent-child hierarchies to handle modular hardpoint adapters.
4. **`ship_default_loadouts`**: A junction table linking a ship's ports to the specific component installed by default.
5. **`file_sync_metadata`**: Tracks file synchronizations for incremental updates (used by the PostgreSQL parser).

## 📋 Prerequisites & Configuration

Ensure you have Python 3.x installed.

### 📦 Game Data Dependency
> [!IMPORTANT]
> For this project to function, it must contain a `libs` folder at the root directory. This folder is generated by extracting the game records using the `starbreaker-datacore` feature from the [StarBreaker tool repository](https://github.com/diogotr7/StarBreaker).
> 
> The database builder scripts expect the parsed JSON files to reside under the structure `libs/foundry/records/` (e.g. `libs/foundry/records/entities/`).

### PostgreSQL Setup
To use PostgreSQL sync, install `psycopg2` and establish a `.env` file in the root workspace directory containing your database credentials:

```ini
DB_HOST=localhost
DB_PORT=5432
DB_NAME=guild_db
DB_USER=admin
DB_PASSWORD=choose_a_strong_password
```

## 🛠️ Data Synchronization & Filtering Rules

To keep the database clean and usable, several filtering guidelines are applied during script runs:

* **Showroom tags**: Only parses ships matching display definitions in `entities/scitem/shopdisplays/vehicle/*.json`. NPC-exclusive ships (e.g., `anvl_arrow_pu_ai_crim`) do not have display pedestals and are filtered out.
* **Component Whitelist**: Only parses components of the following types:
  * `PowerPlant` (Power Plants)
  * `Shield` (Shield Generators)
  * `QuantumDrive` (Quantum Drives)
  * `Cooler` (Coolers)
  * `Radar` (Radars)
  * `WeaponGun` (Weapons)
  * `Missile` (Missiles & Torpedoes)
  * `MissileLauncher` (Missile Racks)
* **Placeholder Stripping**: Skips any component listing name tags containing `loc_placeholder` (case-insensitive) to prevent template assets (e.g., `COOL_S01_Template`) from dirtying database tables.
* **Grade Translation**: Maps raw numeric values of item grades (1, 2, 3, 4) to letters (A, B, C, D).
